# CatVox MVP User Test Plan

## 1. Objective

Run a lightweight early-stage user test for the CatVox MVP to validate:

1. Whether the app's purpose is understandable without heavy explanation.
2. Whether a new user can navigate the app naturally without guidance.
3. Where the user feels confused, blocked, frustrated, or pleasantly surprised.
4. Whether the user sees any real value in using the app again.

This is a pretotyping / customer development session, not a polished usability study.

---

## 2. Roles

### 2.1 Facilitator (me)
The facilitator:

1. Briefly introduces the session.
2. Starts screen recording with microphone enabled.
3. Observes silently during the session.
4. Does not explain features or rescue the user.
5. Asks the 3 post-test questions after the user finishes.
6. Saves the recording and later analyses it with Gemini.

### 2.2 Test user
The test user:

1. Uses the app naturally as a first-time user.
2. Thinks aloud during the whole session.
3. Says out loud what they expect, notice, like, dislike, or find confusing.
4. Completes the session without guidance.
5. Answers the 3 post-test questions at the end.

---

## 3. Session setup

Before the session starts, the facilitator will:

1. Ensure the CatVox app is installed and ready on the iPhone.
2. Ensure screen recording is available on the device.
3. Enable microphone recording for the screen recording.
4. Check that the device has enough battery and storage.
5. Close unrelated apps and notifications if possible, to reduce distractions.
6. Prepare a quiet environment so the user's spoken comments are clearly recorded.
7. Make sure the user knows the screen and voice will be recorded for product learning.

---

## 4. Exact introduction to say to the user

The facilitator will say the following word for word before starting:

> Thanks for trying this. This is an early version of a new app, and I'd like you to use it as naturally as possible. Please imagine you've just opened it because you want to understand what your cat is trying to communicate. As you go, please say out loud what you expect, what you notice, and anything that feels confusing, frustrating, or surprisingly good. I'll stay quiet and won't guide you, because I'm testing the app, not you. I'm going to record the screen and your voice comments so I can review the session afterwards.

---

## 5. Start of test

The facilitator:
1. asks the user if they are ready.
2. starts the iPhone screen recording with microphone enabled.
3. hands the device to the user.
4. remains silent from that point onward unless there is a technical issue.

---

## 6. Rules during the test

### 6.1 Facilitator rules

During the test, the facilitator will:

1. Stay quiet.
2. Not explain the interface.
3. Not suggest what to tap.
4. Not correct misunderstandings.
5. Not defend the app.
6. Only intervene if there is a technical problem that makes the test impossible to continue, and clearly say on microphone what I'm doing to fix to help user continue.
7. Observe the user's actions, pauses, hesitation, facial reactions, and spoken comments.

### 6.2 User instructions during the test

The user should:

1. Explore the app naturally.
2. Try to achieve the goal implied by the scenario.
3. Keep talking throughout the session.
4. Say things such as:
   1. What they think will happen next.
   2. What they believe a button or screen means.
   3. What feels easy.
   4. What feels confusing or frustrating.
   5. What feels useful, interesting, or enjoyable.

---

## 7. Main task scenario

The scenario for the session is:

1. The user has just opened the app.
2. The user wants to understand what their cat is trying to communicate.
3. The user should use the app freely from that starting point.

No additional feature explanation should be given before the session begins.

---

## 8. End of test

The test ends when one of the following happens:

1. The user naturally finishes exploring the app.
2. The user says they are done.
3. The user reaches a clear stopping point.
4. The session becomes repetitive and no longer produces new observations.

Then the facilitator will:

1. Thank the user.
2. Immediately ask the 3 post-test questions.
3. Make sure their answers are also recorded in the same video/voice screen recording session.
4. Speak his own observations (see section 11 below).
5. Stop the screen recording.

---

## 9. Post-test questions

The facilitator will ask these 3 questions exactly:

1. What do you think this app is for, in your own words?
2. What was the most confusing, awkward, or frustrating moment?
3. If this app already existed on your phone, in what situation would you actually use it - and would you come back to it?

---

## 10. What the facilitator should capture during the session

While observing, the facilitator should pay attention to:

1. The first thing the user taps.
2. Which paths or screens they use.
3. Where they hesitate.
4. Where they misunderstand the app.
5. Where they backtrack.
6. Where they stop talking.
7. Any spontaneous positive reaction.
8. Any sign of delight, curiosity, or emotional connection.
9. Any explicit statement of confusion, value, disappointment, or desire.
10. Whether they reach an outcome that feels meaningful to them.

---

## 11. Immediate notes after each session

As the last step in recording session, before deep analysis, the facilitator should speak on the same microphone short notes about:

1. His first impression of what happened.
2. The biggest confusion point.
3. The strongest positive reaction.
4. Whether the user seemed to understand the product value.
5. Whether the facilitator believe this person would realistically use it again.
6. Any quotes worth preserving verbatim.

---

## 12. Recording management

After the session, the facilitator will:

1. Save the screen recording.
2. Give it a clear file name.
3. Store it in a consistent location for later review.

Suggested naming format:

`catvox_test_YYYY-MM-DD_userNN.mov`

Example:

`catvox_test_2026-04-24_user01.mov`

---

## 13. Gemini analysis plan

When reviewing the recording with Gemini, the facilitator should ask it to extract and classify:

1. User journey path.
2. Key taps and navigation decisions.
3. Moments of hesitation.
4. Moments of confusion.
5. Moments of frustration.
6. Moments of delight.
7. Explicit likes.
8. Explicit dislikes.
9. Verbalized expectations.
10. Verbalized value statements.
11. Points where the user appeared lost or nearly abandoned the flow.
12. A short summary of the most important product insights.
13. A separate section with facilitator's observations.

A useful classification model is:

1. Expectation
2. Hesitation
3. Confusion
4. Frustration
5. Delight
6. Value signal
7. Abandonment risk

---

## 14. Recommended test discipline across multiple users

To keep sessions comparable, the facilitator should:

1. Use the same introduction every time.
2. Use the same scenario every time.
3. Stay equally silent in each session.
4. Ask the same 3 end questions every time.
5. Avoid explaining new features differently to different users.
6. Record every session in the same way.
7. Keep a simple summary sheet across all users.

---

## 15. Success criteria for this phase

This test phase is successful if it helps product team learn:

1. Whether users understand what CatVox is for.
2. Whether they can use it without being taught.
3. What parts of the experience break trust, clarity, or interest.
4. Whether any users express a believable reason to use it again.
5. Which product issues should be fixed before broader exposure.

---

## 16. One-page execution checklist

### Before the user arrives

1. App installed and working.
2. Device charged.
3. Enough storage available.
4. Screen recording ready.
5. Microphone enabled for screen recording.
6. Quiet environment prepared.
7. Recording consent / awareness confirmed.

### At the start

1. Say the exact introduction.
2. Confirm the user is ready.
3. Start recording.
4. Hand over the device.

### During the session

1. Stay silent.
2. Observe carefully.
3. Do not guide.
4. Only intervene for technical failure.

### At the end

1. Stop recording.
2. Thank the user.
3. Ask the 3 post-test questions.
4. Save the video with a consistent file name.
5. Write immediate notes.
6. Later analyse the recording with Gemini.

---

## 17. Important reminder

The goal of this session is not to hear polite praise.

The goal is to discover:

1. What the user thinks the app is.
2. What the user actually does.
3. Where reality differs from my assumptions.
4. Whether there is an authentic signal of value worth pursuing.
