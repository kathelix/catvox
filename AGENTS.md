Create a new ADR in `docs/adr/` for any decision that is architectural, security-relevant, infrastructure-relevant, costly to reverse, or likely to need explanation later. See `docs/adr/README.md` for the full convention.

When an ADR is accepted:
- check whether HLD needs updating (high-level direction)
- check whether TRD needs updating (implementation requirements)
- reference the ADR from TRD where relevant (`See ADR-NNNN.`)

## 4. iOS Development

### XcodeGen Workflow

**Never edit `.xcodeproj` directly.** The project file is generated from `project.yml`.

```bash
# After any change to project.yml:
xcodegen generate
```

The `schemes:` block in `project.yml` is intentional — XcodeGen silently deletes shared schemes unless they are declared there. Do not remove it.

### Key Implementation Details

- **Video recording:** `CameraService.swift` — `sessionPreset = .hd1920x1080` caps at 1080p; HEVC codec enforced via `setOutputSettings` after `addOutput()`. Falls back to H.264 silently on pre-A10 devices.
- **Mock mode:** `GCPService.swift` has `mockMode = true` by default. The real upload pipeline is stubbed until the backend is deployed. Flip to `false` once the Cloud Function is live.
- **Confidence score color coding:** Result screen ring is green (>80%), amber (50–80%), red (<50%) — implemented in ResultViewModel.
- **Share watermark:** "Powered by Kathelix" (no ".com") on free-tier exports.

### Vertex AI Call Parameters (for Cloud Function implementation)

Implemented in `functions/src/gemini.ts` — update there, not here:
- **Temperature:** 0.7
- **Response MIME type:** `application/json`
- **Safety settings:** `BLOCK_ONLY_HIGH` (standard violence filters trip on feline hunting behavior)

## 5. Infrastructure

### Terraform State

Remote state lives in GCS: `gs://catvox-tf-state-kathelix-catvox-prod/catvox/state`

Never run `terraform apply` locally without first confirming the remote state is clean:
```bash
cd terraform
terraform init   # connects to GCS backend
terraform plan   # review before any apply
```

`terraform.tfvars` is gitignored. Copy from `terraform.tfvars.example` and fill in values.

### Service Accounts

Two SAs with strictly separated concerns (TRD §6.3):

| SA | Purpose | Key roles |
|---|---|---|
| `catvox-backend-sa` | Cloud Functions runtime | aiplatform.user, storage.objectViewer, datastore.user, secretmanager.secretAccessor, iam.serviceAccountTokenCreator (self) |
| `catvox-ci-sa` | GitHub Actions Terraform CI | roles/editor, roles/resourcemanager.projectIamAdmin, storage.objectAdmin on state bucket |

Do not add CI roles to `catvox-backend-sa` — the split is intentional. If a Cloud Function were compromised, `catvox-backend-sa` having CI roles would give an attacker near-full project control.

### WIF / GitHub Actions Authentication

Keyless auth via Workload Identity Federation — no long-lived service account keys anywhere. GitHub Actions presents an OIDC token; GCP exchanges it for a short-lived credential scoped to `catvox-ci-sa` (the dedicated Terraform CI identity).

The WIF pool is locked to `IvanBoyko/catvox` via `attribute-condition` — no other repo can impersonate the SA.

### CI/CD Pipelines

| Workflow | Trigger | What it does |
|---|---|---|
| `build.yml` | Every push/PR to `main` | XcodeGen → build for iOS Simulator (no signing) |
| `terraform.yml` (plan) | PR touching `terraform/**` | fmt-check → init → validate → plan → PR comment |
| `terraform.yml` (apply) | Merge to `main` touching `terraform/**` | init → apply -auto-approve |

### Bootstrap (one-time, already done)

Both bootstrap scripts in `terraform/` have already been run against the live project. They are idempotent — safe to re-run if needed:

```bash
PROJECT_ID=kathelix-catvox-prod ./terraform/bootstrap_remote_state.sh
PROJECT_ID=kathelix-catvox-prod ./terraform/bootstrap_wif.sh
```

**Required GitHub Actions secrets** (already configured):

| Secret | Value |
|---|---|
| `GCP_PROJECT_ID` | `kathelix-catvox-prod` |
| `GCP_WORKLOAD_IDENTITY_PROVIDER` | Full WIF provider resource name |
| `GCP_SERVICE_ACCOUNT` | `catvox-ci-sa@kathelix-catvox-prod.iam.gserviceaccount.com` |
| `TF_VAR_app_check_debug_token` | Firebase App Check debug token |

## 6. Process Conventions

### Branching

- Feature work goes on a branch; open a PR to `main`.
- The Terraform pipeline posts a plan comment on every PR — review it before merging.
- Use descriptive branch names: `feature/`, `fix/`, `infra/`.

### Commit Style

Short, imperative subject line. No period. Examples from this repo's history:
```
Add GoogleService-Info.plist, persist scheme via project.yml
TRD v1.8 — add CI/CD section, fix six audit findings
```

### TRD Sync Rule

Any time you implement something from TRD §8 backlog, mark it `[x]` in TRD immediately — in the same commit as the implementation. The TRD is the source of truth for implementation status.

### HLD vs TRD

- **docs/HLD.md** — only stable design decisions (the "why"). No implementation status, no concrete values that will change.
- **docs/TRD.md** — everything concrete: versions, bucket names, roles, backlog status.
- When you want to change a design decision, update HLD first, then TRD, then code.

## 8. Key Constraints and Decisions Not Obvious from Code

- **No ads.** Monetization is StoreKit 2 IAP only (from `docs/PROMPT.md`).
- **Firebase as backend proxy.** The iOS app never calls Vertex AI directly — all AI calls go through a Cloud Function. This is a security requirement (App Check + cost control).
- **HEVC fallback is silent.** Pre-A10 devices fall back to H.264 without any UI change. This is intentional.
- **GCS video lifecycle is 1 day.** Clips are deleted after 24 hours by a GCS lifecycle rule — privacy by design.
- **State bucket is outside Terraform.** It cannot manage its own state backend — bootstrapped manually to avoid circular dependency.
- **`mockMode = true` is the safe default.** Any new developer running the app locally will get mock AI responses. Real backend requires the Cloud Function to be deployed first.
